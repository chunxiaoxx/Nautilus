### 2. 技术架构问题

#### Q4: 为什么选择 WebSocket 而不是 HTTP？
**A**: WebSocket 的优势：
- ✅ **双向实时通信**: 智能体间需要持续的双向通信
- ✅ **低延迟**: 实时协作需要快速响应
- ✅ **连接保持**: 避免频繁建立连接的开销
- ✅ **推送能力**: 服务器可以主动推送消息给智能体

HTTP 适合请求-响应模式，但智能体协作需要持续的双向通信。

---

#### Q5: Nexus Protocol 的核心设计理念是什么？
**A**: 核心设计理念：
1. **简单性**: 10种消息类型覆盖完整的 A2A 通信
2. **可靠性**: ACK/NACK 确认机制保证消息送达
3. **安全性**: HMAC-SHA256 消息签名验证
4. **灵活性**: 支持单播、广播、组播
5. **健壮性**: 并发控制和消息过期机制

---

#### Q6: 如何保证消息的可靠性？
**A**: 多层保障机制：
1. **ACK/NACK 确认**: 接收方确认消息收到
2. **消息签名**: HMAC-SHA256 防止篡改
3. **消息过期**: TTL 机制避免过期消息
4. **并发控制**: 队列限制防止资源耗尽
5. **错误处理**: 完善的错误统计和日志

### 3. 实现细节问题

#### Q7: 10种消息类型分别是什么？
**A**:
1. **HELLO** - 智能体注册
2. **REQUEST** - 请求协作
3. **OFFER** - 提供能力
4. **ACCEPT** - 接受请求
5. **REJECT** - 拒绝请求
6. **PROGRESS** - 进度更新
7. **COMPLETE** - 任务完成
8. **SHARE** - 知识共享
9. **ACK** - 消息确认 ✨
10. **NACK** - 消息拒绝 ✨

(ACK/NACK 是 Day 3 新增的)

---

#### Q8: 并发控制是如何实现的？
**A**: 两层控制机制：
1. **消息队列限制**: 默认 1000 条消息
   - 队列满时拒绝新消息
   - 返回 QUEUE_FULL 错误

2. **智能体数量限制**: 默认 100 个智能体
   - 连接数满时拒绝新智能体
   - 返回 MAX_AGENTS_REACHED 错误

可配置：`NexusServer(max_queue_size=500, max_agents=50)`

---

#### Q9: 消息过期机制是如何工作的？
**A**: 支持两种过期方式：
1. **TTL (相对时间)**:
   ```python
   msg = NexusMessage(..., ttl=60)  # 60秒后过期
   ```

2. **expires_at (绝对时间)**:
   ```python
   msg = NexusMessage(..., expires_at=datetime.now(UTC) + timedelta(hours=1))
   ```

服务器在处理消息前自动检查是否过期，过期消息会被拒绝。
